## 3. Design Philosophy

> Ask before every decision: *"Would a line cook understand this at a glance, with wet hands, mid-service?"*

- **Clarity over cleverness** — status must be legible instantly. No ambiguity.
- **Touch-first** — minimum 44px touch targets (WCAG 2.5.5). Assume finger, not mouse, on iPad.
- **Colour as signal, not decoration** — green = IK-Mat/safe, amber = IK-Alkohol/caution, red = alert/action required.
- **Minimal navigation on iPad** — never more than 2 taps to complete a primary task.
- **No half-finished features ship** — if something isn't complete and polished, it's hidden behind a feature flag or removed before submission.

## 4. CSS Theming System

All module-specific colours derive from a single hue variable. Changing the hue changes all shades consistently.

```css
/* src/assets/main.css — one line to change per module */
:root {
  --ik-mat-hue:     142;   /* green */
  --ik-alkohol-hue:  38;   /* amber — change to 142 for green, 271 for purple, etc. */
}

.module-ik-mat {
  --mod-bg:      hsl(var(--ik-mat-hue), 100%, 97%);
  --mod-border:  hsl(var(--ik-mat-hue), 60%,  75%);
  --mod-accent:  hsl(var(--ik-mat-hue), 70%,  40%);
  --mod-text:    hsl(var(--ik-mat-hue), 80%,  20%);
  --mod-dark:    hsl(var(--ik-mat-hue), 90%,  12%);
}

.module-ik-alkohol {
  --mod-bg:      hsl(var(--ik-alkohol-hue), 100%, 97%);
  --mod-border:  hsl(var(--ik-alkohol-hue), 90%,  70%);
  --mod-accent:  hsl(var(--ik-alkohol-hue), 85%,  45%);
  --mod-text:    hsl(var(--ik-alkohol-hue), 80%,  25%);
  --mod-dark:    hsl(var(--ik-alkohol-hue), 90%,  15%);
}
```

**Rule:** Every IK-Alkohol-specific colour in components must use `var(--mod-*)` tokens, never hardcoded amber hex values. This guarantees the theme is changeable from one line.

## 5. iPad Layout

### 5.1 Tab Bar Structure

5 tabs. Never more. Less-used views go behind "Mer" overflow.

| Position | Tab | Always visible |
|----------|-----|----------------|
| 1 | 🏠 Hjem | Yes |
| 2 | 🌡️ Temp | Yes |
| **3 (centre)** | **✅ SJEKK** — raised green primary button | Yes |
| 4 | ⚠️ Avvik | Yes |
| 5 | ☰ Mer | Yes — opens drawer: Grafer, Opplæring, Alkohol, Innstillinger |

The centre tab is visually raised (green rounded rect, larger icon, white label) — primary action at a glance.

**This tab bar is iPad-only.** Desktop uses the full sidebar (unchanged structure, with IK-Alkohol section added — see section 9).

### 5.2 Overflow Drawer ("Mer") — iPad only

Tapping ☰ opens a bottom sheet with large tap targets for secondary navigation:
- Grafer
- Opplæring
- IK-Alkohol (navigates to alkohol module home — see section 7)
- Innstillinger (admin only — hidden for STAFF/MANAGER without settings permission)

### 5.3 IK-Alkohol navigation on iPad

The main 5-tab bar does **not** change when entering IK-Alkohol. The tab bar is global navigation. Instead, IK-Alkohol uses internal horizontal sub-tabs at the top of the view (same pattern as the existing settings tabs):

```
[ Alderskontroll ] [ Sjekklister ] [ Hendelseslogg ]
```

An amber module banner sits above these sub-tabs to clearly signal which module you're in. This way Avvik, Temp, and Sjekk remain accessible from the bottom tab bar at all times.

## 6. Temperature Logging UX

### 6.1 Unit Cards

The temperature tab shows a 2-column grid of unit cards. Each card has four states:

| State | Visual |
|-------|--------|
| Not logged today | White card, blue CTA "Trykk for å logge →" |
| Logged, in range | Green border + background |
| Logged, out of range — no deviation filed | Red border + background, persists until deviation is created |
| Logged, deviation filed + in progress | Amber border |

### 6.2 Inline Logging Flow

Tapping a card:
1. Card gets thick indigo left border (5px), bottom-left radius collapses to 0
2. An input row appears below spanning full width, with matching indigo left border and flat top-left corner — creating one continuous visual left edge connecting card to input
3. Input row contains: temperature number field (large) + °C label + optional note field + ✕ cancel
4. Full-width green **Lagre** button on its own row below (min-height 44px, WCAG 2.5.5)

No "Neste" button — staff choose which unit to log next by tapping its card. No assumed order.

**Future enhancement (to try, not in initial implementation):** SVG/canvas overlay draws a thick line from the active card's bounding box to the input box position using `getBoundingClientRect()`. Fallback is the CSS left-border approach.

### 6.3 Post-save behaviour

After Lagre:
- Card immediately updates to green (or red if out of range) — optimistic UI
- Input row collapses with a smooth transition
- If reading is out of range: red card shows a "Rapporter avvik?" prompt

## 7. IK-Alkohol Module

Full scope — 3 screens + dashboard section + module-aware reports.

### 7.1 Module Identity

- Amber colour temperature (`--ik-alkohol-hue: 38`)
- Module banner at top of every IK-Alkohol screen: `[amber dot] IK-Alkohol`
- Sidebar shows separate section heading "IK-Alkohol" below "IK-Mat" items
- iPad: active tab highlights amber instead of green

### 7.2 Screen 1 — Alderskontroll-logg (Age Verification Log)

**Purpose:** Log every ID check performed when serving alcohol.

**Fields per entry:**
- Timestamp (auto-filled, editable)
- Staff member (auto-filled from logged-in user)
- Outcome: `Godkjent` (OK) / `Nektet` (Denied) / `Usikker` (Unsure)
- Optional note

**UI:** Same inline card pattern as temperature logging. Each log entry is a card showing time, staff, outcome badge (green/red/amber), and note. New entry form is a simple inline form at the top.

**Backend needed:** `GET /api/alkohol/age-verifications`, `POST /api/alkohol/age-verifications`

### 7.3 Screen 2 — Alkohol-sjekklister

**Purpose:** Opening/closing bar routines specific to alcohol compliance.

**Implementation:** Reuses the existing checklist engine entirely. `moduleType: 'IK_ALKOHOL'` on the checklist instances. The only difference is the amber colour theme and the module banner.

No new components needed — `ChecklistView.vue` accepts a `module` prop and applies the correct theme class.

**Backend needed:** Existing checklist endpoints with `moduleType=IK_ALKOHOL` filter.

### 7.4 Screen 3 — Hendelseslogg (Incident Log)

**Purpose:** Log alcohol-related incidents: refused service, intoxicated guest, police contact.

**Fields per incident:**
- Incident type: `Nektet servering` / `Beruset gjest` / `Politikontakt` / `Annet`
- Description (required)
- Time (auto-filled, editable)
- Staff on duty (auto-filled)
- Follow-up required: yes/no toggle
- Linked deviation (optional — creates a deviation in the main deviations system)

**UI:** Similar to DeviationsView but scoped to IK-Alkohol. Incident cards show type badge, time, staff, and follow-up status.

**Backend needed:** `GET /api/alkohol/incidents`, `POST /api/alkohol/incidents`

### 7.5 Dashboard Section

An IK-Alkohol summary block appears on the main Dashboard when the module is enabled:
- ID checks today: N
- Incidents this week: N
- Alkohol-sjekkliste completion %
- Quick-link to alderskontroll-logg

Hidden if `organization.modules.ikAlkohol === false`.

## 8. Permission Gating

Two layers, implemented in UI:

### 8.1 Route-level (already implemented)
- `ADMIN` only: `/innstillinger/**`
- All authenticated: everything else
- Enforced in `router/index.ts` guard — already done

### 8.2 Feature-level (to implement)
Read `useAuthStore().user.permissions` and conditionally render.

| Permission flag | What it hides when false |
|-----------------|--------------------------|
| `temperatureLogging` | Log-temp button on unit cards |
| `checklists` | Checklist completion controls (view-only mode) |
| `reports` | Export buttons, Grafer nav item |
| `deviations` | "Rapporter avvik" button |
| `userAdmin` | Users tab in settings |
| `settings` | Settings nav item entirely |

**Pattern:** Use a composable `usePermission()` that reads from auth store.

```typescript
// src/composables/usePermission.ts
export function usePermission() {
  const auth = useAuthStore()
  const can = (key: keyof UserPermissions) =>
    auth.user?.permissions?.[key] ?? false
  return { can }
}
```

Usage in template: `v-if="can('temperatureLogging')"`. No prop drilling.

## 14. WCAG Requirements

Audit and fix across all components:

| Requirement | Action |
|-------------|--------|
| Semantic HTML | Replace `<div>` buttons with `<button>`, use `<nav>`, `<main>`, `<section>`, `<article>` |
| `aria-label` | All icon-only buttons need labels (e.g., sidebar collapse, close modal) |
| Keyboard nav | Tab order logical, Enter/Space activate buttons, Escape closes modals/drawers |
| Focus ring | Visible `:focus-visible` ring on all interactive elements — 2px solid, offset |
| Colour contrast | Check all text against backgrounds — WCAG AA (4.5:1 normal, 3:1 large) |
| Status not by colour alone | Every badge/status also has text (not just red/green) |
| Touch targets | Min 44×44px for all interactive elements |

## 18. Design Decisions Log

| Decision | Choice | Reason |
|----------|--------|--------|
| iPad tab count | 5 tabs | More crowds touch targets |
| iPad primary tab | Sjekklister (centre, raised) | Primary daily action for staff |
| Temperature logging | Unit cards + inline input | Halves tap count vs modal approach |
| "Neste" button | Removed | Assumes order; staff log selectively |
| Lagre button | Full-width, own row | Never overflows; 44px+ touch target |
| Card-to-input connection | Matching 5px indigo left border | CSS-only, no JS, clear visual link |
| SVG connector line | Future enhancement to try | Richer UX but not in initial build |
| IK-Alkohol colour | Amber via CSS hue variable | Instant module recognition |
| Module hue variable | Single `--ik-alkohol-hue` int | Change one value, all shades follow |
| Permission gating | `usePermission()` composable | No prop drilling, reusable |
| IK-Alkohol nav | Own sidebar section + amber active | Compliance clarity — wrong module = wrong log |
